# W-Model 左辺レビュー: フェーズゲート設計と検証項目数管理の学び

## 何が起きたか

5ファイル・5要件の Markdown 変更で、W-Model の左辺レビュー（設計フェーズでの並行検証活動）を導入した。具体的には、(1) /brainstorm にテスタビリティレビュー、(2) reference/core-rules.md にフェーズゲート、(3) /spec と /implement からのゲート参照、(4) /implement に中間レビューポイント、(5) spec-validator にエントリー基準プレチェックを追加した。

レビュー指摘は P3（軽微な観察事項）3件のみで、P1/P2 はゼロ。横断整合性チェックも全項目 PASS。

## 教訓

### 1. 検証項目数の変更を避ける設計が最も安全

spec-validator の検証項目数（「11 の検証項目」）は commands/spec.md と spec-validator.md の2箇所で言及されている。Phase 1（add-traceability）では「9→10」、Phase 2（test-multilayer）では「10→11」の変更時にそれぞれ同期漏れが発生した。

今回はエントリー基準チェックを「Step 0（前提条件チェック）」として検証項目とは独立させ、数値変更を回避した。この設計により同期リスクが完全に消滅した。

**原則**: 既存の「N個の〇〇」という数値記述に影響する変更は、可能な限り独立した概念として分離する。

### 2. 宣言的チェックリスト > 自動ブロッキング

Phase Gates を「宣言的なチェックリスト」として設計し、自動ブロッキングにしなかった。これにより:
- 実装の複雑さが最小限（Markdown への記述追加のみ）
- ゲートオーバーヘッドの落とし穴（ゲート渋滞、形式主義化）を回避
- 将来的な自動化への拡張パスは維持（spec-validator Step 0 が自動検証の先行事例）

### 3. 概念の分離命名が重要

「Verification Gates」（テスト実行レベル）と「Phase Gates」（ワークフロー遷移基準）は同じファイル内に併存するが、異なる概念である。セクション見出しと説明文で明確に区別したことで、横断チェックでの混同がなかった。

### 4. シンボリックリンクによる同期は信頼できる

過去の compound learnings で繰り返し報告されていた「プロジェクト/グローバル同期漏れ」問題は、シンボリックリンク方式により完全に解消されている。横断チェックで確認済み。

## 防止策

- [x] spec-validator の検証項目追加時は「Step 0」パターン（前提条件として分離）を検討する
- [x] 同一ファイル内に異なる概念のセクションを追加する場合、命名で明確に区別する
- [x] フェーズゲートは宣言的チェックリストから始め、必要に応じて自動化する
